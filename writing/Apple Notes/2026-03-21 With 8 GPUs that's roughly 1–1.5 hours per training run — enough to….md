With 8 GPUs that's roughly 1–1.5 hours per training run — enough to run maybe 8–10 experiments in a day. The right order is eval-side first (minutes, no retraining), then training changes ranked by expected impact.
**Immediate, no retraining:**
Sequential retrieval is the highest-value thing to test right now. It doesn't require touching the model at all — just add a --rerank_n argument to retrieval_eval.py: use the semantic axis to get the top-N candidates, then re-rank that shortlist by the combined score. With N=50 or N=100, the speaker axis can only influence items that are already semantically close. This is almost certainly better than the current approach and takes 10 minutes to implement and test.
**Training experiments worth running, in order:**
1. **Retrain with 256-dim semantic projection** (restore original hub model dims: semantic:256 speaker_id:128 dialect:64 gender:32). The hub model reportedly did better on rehasp. Confirm it cleanly with the new eval pipeline.
2. **Gradient reversal on the speaker head.** Add a GradientReversalLayer before the speaker projection so the backbone is penalized for encoding speaker identity. Standard domain adaptation trick, directly addresses the root failure. ~30 lines of code in train_wavlm.py.
3. **Held-out speaker validation.** Use one speaker not in training (not p315) as a validation set during training instead of OSR in-distribution eval. This would have caught the generalisation failure during training. Combined with early stopping this could improve generalisation directly.
4. **Axis loss weighting.** Currently all axes contribute equally to the total loss. Try downweighting speaker loss relative to semantic during training — force the backbone to prioritize content over speaker identity.
